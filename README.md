# Media Feed

A cross-platform short-video feed (TikTok-style) sharing one codebase across
**Web**, **Android**, and **Fire TV**, built with React Native + React Native Web.

## Stack

- **React Native** ‚Äî Android / Fire TV (Fire TV apps are Android apps; the
  same `react-native` Android target ships to both, with TV-specific
  navigation/focus handling layered on top as the app grows).
- **React Native Web** ‚Äî renders the same components to the DOM for the
  browser.
- **Vite** ‚Äî dev server + bundler for the web target only. Native targets
  still run through Metro (RN's own bundler) via the standard RN CLI ‚Äî Vite
  does not replace Metro for native builds.
- **TypeScript** ‚Äî shared across all platforms.

## Getting started

```bash
npm install
```

### Web

```bash
npm run web          # dev server
npm run build:web    # type-check + production build
npm run preview:web  # preview the production build
```

### Android / Fire TV

This package.json gives you the JS dependencies only. Native Android/Fire TV
builds need a native project, which isn't generated by `npm install` alone.
If you don't already have an `android/` folder, scaffold one with the
React Native Community CLI:

```bash
npx @react-native-community/cli@latest init TmpScaffold --skip-install
# then copy the generated android/ folder into this project
```

Once `android/` exists:

```bash
npm run start     # Metro bundler
npm run android    # build + run on a connected device/emulator (incl. Fire TV via adb)
```

For Fire TV specifically, sideload via `adb connect <firetv-ip>:5555` and
`npm run android` as you would for any Android device ‚Äî Amazon's Fire TV
accepts standard Android (`.apk`) builds.

**Why feature folders, not type folders:** grouping by domain (`feed/`,
later `auth/`, `profile/`, etc.) instead of by type (`components/`,
`screens/`, `hooks/`) keeps related files together and scales better as
the app grows. Add a `components/` subfolder *inside* a feature only once
that feature needs more than one supporting component.

As the app grows, you'll likely add:

```

```

## Notes

- Safe-area handling uses `react-native-safe-area-context` rather than core
  RN's `SafeAreaView`, which is deprecated. `SafeAreaProvider` must wrap the
  app root ‚Äî this is required for correct behavior on web, not just a nice-to-have.
- `FeedItem.tsx` is a placeholder ‚Äî swap the `<Text>` content area for an
  actual video component once you pick a player. For HLS streaming, a
  library like `react-native-video` works on Android; for web you'd pair it
  with `hls.js`, since native `<video>` doesn't support HLS in most browsers.
- **Flow-type stripping is handled by a small custom Vite/esbuild plugin
  pair in `vite.config.ts`, not `vite-plugin-react-native-web`.** That
  package ships a CommonJS file under an ESM `package.json`
  (`"type": "module"`), which crashes immediately when Vite's config loader
  tries to `require()` it (`ReferenceError: exports is not defined`). Our
  replacement does the same job ‚Äî stripping Flow syntax from `react-native`
  / `react-native-web` source files before esbuild parses them ‚Äî via two
  small plugins: one for the normal dev/build transform pipeline, one for
  Vite's separate dependency pre-bundling pass (which runs its own esbuild
  call and doesn't share config with the main pipeline). This also needed:
  - `resolve.extensions` AND `optimizeDeps.esbuildOptions.resolveExtensions`
    set to the same list ‚Äî these are two separate options and Vite does not
    apply the first to dependency pre-bundling automatically. Without the
    second, `.web.js` platform-specific files in packages like
    `react-native-safe-area-context` get skipped in favor of native-only
    `.js` files that reference RN internals with no web equivalent.
  - An exact-match (not prefix) alias for `react-native` ‚Üí `react-native-web`,
    since a naive string-prefix alias breaks subpath imports like
    `react-native/Libraries/...`.
  - `define` entries for `global`, `__DEV__`, and `process.env.NODE_ENV`,
    since RN packages assume a webpack-style environment that defines these
    globally by default.
  - If hardware acceleration fails use `sudo modprobe kvm_amd` to load module
    and `lsmod | grep kvm` to verify
